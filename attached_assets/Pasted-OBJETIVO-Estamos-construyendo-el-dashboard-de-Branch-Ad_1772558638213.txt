FASE 2: SEGURIDAD AL DESACTIVAR/ELIMINAR PLANES
PROBLEMA: Si desactivo/eliminio un plan que ya está asignado a clientes, debo evitar inconsistencias.
CAMBIO:

- Implementa “Desactivar/Archivar” (soft delete) como comportamiento por defecto.
- Si el usuario intenta “Eliminar”:
  - Si NO tiene clientes asignados: permitir eliminar.
  - Si SÍ tiene clientes asignados: mostrar modal:

- Si se elige desasignar:
  - Para cada cliente afectado, su plan actual debe quedar en “Sin plan asignado”
  - NO borres historial de compras/uso si existe; marca como “plan archivado” o guarda snapshot (planName/price/classes/days) en la asignación.

FASE 3: LÓGICA DE CONSUMO DE CLASES (lo más importante)
NECESIDAD:
- Cuando un cliente tiene un plan con clases (ej 12), se debe descontar 1 clase bajo estas reglas:
  1) Si el cliente agenda y ASISTE => descuenta 1
  2) Si el cliente agenda y NO CANCELA y ya pasó la hora de la clase => descuenta 1 (no-show)
  3) Si el cliente cancela ANTES del tiempo límite => NO descuenta
- Debe verse:
  - Clases restantes y vencimiento en perfil del cliente
  - Alerta/notificación en dashboard cuando un cliente se queda sin clases (0) o se venció su plan

DISEÑO MINIMO (sin romper):
- En la asignación del plan al cliente, guarda:
  - assignedAt (fecha)
  - expiresAt (nullable si sin vencimiento)
  - totalClasses (nullable si ilimitadas)
  - remainingClasses (nullable si ilimitadas)
- En booking/reserva, guarda:
  - status: booked | cancelled | attended | no_show
  - cancelledAt (nullable)
  - source: app | dashboard (para futuro)
- Agrega config por sucursal:
  - cancelCutoffMinutes (default 120) para “cancelar a tiempo”

IMPLEMENTACIÓN:
A) Descuento por ASISTENCIA:
- Cuando se marca “Asistió”, si la reserva estaba booked, cambia a attended y descuenta 1 (si no es ilimitado).
- Evita doble descuento (idempotencia): guarda en la reserva un flag “consumed=true” o detecta por status.

B) Descuento por NO-SHOW (pasó la hora):
- Implementa un job simple (server-side) que al cargar el dashboard o cada X minutos:
  - Busca reservas booked cuyo endTime < now y no están cancelled/attended
  - Márcalas no_show y descuenta 1 (idempotente)
- No uses cron externo; hazlo con un scheduler liviano en el server o un “lazy evaluator” al consultar.

C) Cancelación:
- Al cancelar, calcula si está “a tiempo”:
  - now <= startTime - cancelCutoffMinutes => cancelled (no consume)
  - si cancela tarde: opcional (define comportamiento). Por ahora:
    - si cancela tarde, marcar cancelled_late y consumir 1 (o tratar como no_show). Elige una sola regla consistente y muéstrala en UI.
- IMPORTANTE: si ya se consumió, no regreses clases automáticamente (a menos que admin lo permita).
