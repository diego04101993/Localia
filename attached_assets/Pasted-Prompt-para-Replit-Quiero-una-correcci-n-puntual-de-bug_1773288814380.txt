BUG 1 — Al quitar y volver a asignar un plan, NO se borran las reservas anteriores del cliente
Problema observado

Si a un cliente le quito su plan actual y luego le asigno otro nuevo, las reservas que tenía con el plan anterior siguen apareciendo.
Eso no debe pasar.

Regla correcta

Cuando un cliente:

pierde su plan,

cambia de plan,

o se le reasigna un nuevo ciclo,

se deben limpiar sus reservas futuras pendientes/confirmadas del ciclo anterior, para que el nuevo plan empiece limpio.

Comportamiento esperado

Si quito un plan a un cliente:

eliminar o cancelar automáticamente sus reservas futuras activas (confirmed) asociadas a su membresía/ciclo anterior.

Si después le asigno un nuevo plan:

el cliente empieza con su plan nuevo sin arrastrar reservas del plan anterior.

No borrar historial pasado:

reservas históricas ya asistidas/no_show/canceladas pueden quedarse para auditoría.

solo limpiar futuras activas.

Implementación esperada

Revisar dónde se ejecuta “quitar plan” y “asignar plan”.

Al quitar plan:

cancelar o borrar solo bookings futuras confirmed del cliente en esa sucursal.

Al asignar nuevo plan:

confirmar que el cliente no conserva bookings activas previas.

Si existe relación membershipId en bookings, usarla.

Si no existe, usar:

userId

branchId

status = confirmed

class start >= now

Entregable

Qué query usaste para limpiar reservas futuras

BUG 3 — Si pasa la hora de la clase y el cliente no cancela, NO se procesa automáticamente el no_show ni el descuento
Problema observado

Si una clase ya pasó y el cliente no canceló:

no se marca solo,

no se descuenta la clase,

y tengo que ir manualmente al dashboard a poner “Asistió” o “No asistió”.

Eso está mal.

Regla correcta

Si una reserva está en confirmed y ya pasó la hora de la clase:

el sistema debe procesarla automáticamente.

Si no canceló antes:

cambiar a no_show

descontar 1 clase si aplica

Si ya estaba marcada manualmente como attended, no tocarla.

Nunca duplicar descuentos.

Comportamiento esperado

El procesamiento automático debe ejecutarse también aunque yo no entre a la pestaña de Reservas.

Debe correr cuando el cliente abra su app, o cuando el admin abra dashboard/resumen/clientes/reservas.

Si ya pasó la clase:

el booking deja de ser confirmed

pasa a no_show

se descuenta clase

se refleja en perfil del cliente

baja contador de clases restantes

Mis reservas ya no debe mostrar esa clase como próxima.

Implementación esperada

Ya existe lógica de reconciliación, pero no está impactando bien todos los lugares.

Quiero que hagas esto:

Reusar la reconciliación existente, no crear una segunda lógica distinta.

Ejecutarla también en:

endpoint de agenda pública del cliente

endpoint de mis próximas reservas

endpoints de resumen/alertas/clientes relevantes del dashboard de sucursal

Asegurar idempotencia:

no descontar dos veces

no cambiar si ya está attended o cancelled

Asegurar que si el booking pasa a no_show:

desaparezca de próximas reservas

se refleje en perfil del cliente

Importante

No inventes cron jobs todavía si no existen.
Prefiero usar reconciliación al cargar vistas/endpoints críticos, para no romper nada.
